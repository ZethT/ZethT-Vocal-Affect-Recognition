Feature Extraction

Extraction Methods

Features are computed per audio clip and mean-aggregated across time frames to produce a single 17-dimensional feature vector. The same extraction code runs in both the training notebook and the deployed FastAPI service.

Fundamental Frequency librosa.yin(fmin=50, fmax=500)
Pitch estimated via the YIN autocorrelation algorithm. Valid range restricted to 50–500 Hz to exclude sub-vocal and falsetto artifacts. Mean f0 across voiced frames is used.
Spectral Centroid librosa.feature.spectral_centroid()
Weighted mean frequency of the power spectrum, computed per frame and averaged. Measures the "center of mass" of spectral energy — a proxy for timbral brightness.
Spectral Rolloff librosa.feature.spectral_rolloff()
Frequency below which 85% of spectral energy lies, averaged across frames. Highly correlated with spectral centroid (r = 0.89) — both track brightness.
HNR (Harmonic-to-Noise Ratio) parselmouth → to_harmonicity_cc()
Computed via Praat's cross-correlation harmonicity algorithm (time_step = 0.01 s, min_pitch = 50 Hz). Values below −20 dB are treated as extraction artifacts and excluded. Mean HNR across valid frames is used.
MFCCs (13 coefficients) librosa.feature.mfcc(n_mfcc=13)
Mel-Frequency Cepstral Coefficients computed on a Mel-scaled filterbank. Each coefficient is averaged across frames. Captures the spectral envelope shape at multiple scales of resolution — the feature set that drove accuracy from 70% to 92.8%.
Model Comparison

Classifiers Evaluated

Each model was trained on an 80/20 stratified train-test split (random_state = 42) and evaluated on the held-out test set. All three were tested on both the base four-feature set and the MFCC-augmented 17-feature set.

Deployed ✓

Logistic Regression

Linear classifier predicting class membership as a weighted sum of input features. A StandardScaler is embedded in a scikit-learn Pipeline to prevent data leakage. Used as the interpretable baseline and final deployed model — coefficients directly show each feature's direction and magnitude of influence. Adding MFCCs pushed its accuracy from 58.6% to 91.4% — within 1.4 points of the top-performing XGBoost model, and chosen for deployment over XGBoost for its simplicity and interpretability.

Max Iterations 1000
Scaler StandardScaler
Accuracy (+ MFCC) 91.4%

Random Forest

Ensemble of 100 decision trees, each trained on a random feature and sample subset. Robust to non-linear boundaries and outliers in the HNR distribution (which ranges from −86 to +24 dB). Evaluated with 5-fold stratified cross-validation to estimate generalization. Gini-based feature importance scores identify which features drive classification — key for understanding why MFCCs matter.

n_estimators 100
Cross-validation 5-fold stratified
Accuracy (+ MFCC) 90.0%

XGBoost

Gradient-boosted trees that sequentially correct prior errors. Each new tree focuses on residuals from the previous ensemble. Fitted via XGBClassifier with a LabelEncoder for scikit-learn compatibility. Consistent top-performer on structured tabular data, included to test whether sequential error correction outperforms parallel ensembles on this small, clean feature set.

Metrics Used

Accuracy

Proportion of correctly classified samples on the held-out test set. Primary metric reported for all model variants. Classes are balanced, so accuracy is not misleading here.

Precision & Recall

Per-class precision (of all predicted positives, how many are correct) and recall (of all actual positives, how many were found). Reported via classification_report.

5-Fold Cross-Validation

Applied to Random Forest on the full dataset to estimate generalization variance. Reports mean accuracy and per-fold scores. Used to confirm the test-set result is stable.

ROC-AUC

Area under the receiver operating characteristic curve. Measures discrimination ability across all classification thresholds, independent of the default 0.5 decision boundary. Plotted for both LR and RF.

System Architecture

Deployment Design

The trained model is served as a containerized FastAPI application. Prediction metadata is persisted to Firestore. The static frontend is hosted separately on App Engine and communicates with the inference service via fetch().
